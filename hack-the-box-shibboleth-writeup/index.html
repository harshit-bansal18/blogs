110.10.11.124 monitor.shibboleth.htb monitoring.shibboleth.htb zabbix.shibboleth.htb shibboleth.htb

Now jump into the browser to explore these subdomains. On opening them, I found that all three are actually the same, and it could also be realized earlier by seeing that all three return same size and words. Anyway, what we have got is a Zabbix monitoring service. On searching for zabbix on google, I found several vulnerabilities like cross-site scripting, zabbix_ID cookie exploit etc. in different versions of the service. But the problem was I couldn't find the version of zabbix this site was using to this led to nowhere. After some waste attempts, UDP port struck me. It is common thing to overlook UDP ports sometimes.

UDP Port Scan

I used nmap to scan the UDP ports. It took long time to perform scan on all the ports, but I got something :).

Exploiting IPMI

I searched for asf-rpmc service and luckily, third link talked about IPMI service exploit.

using msfconsole, I found that the service was using Ciper-0 suite. I started with exploiting Ciper-0 but realized that in the end I would require ssh servie which is not available on this machine. So I looked on next link which exploited the service by grabbing hashes using ipmi_dumphahes module.

1msfconsole
2use auxiliary/scanner/ipmi/ipmi_dumphahes
3set RHOSTS 10.10.11.124
4run

Very quickly, I got the hash.

Post Authentication Exploit

After logging in, we get a very complex website, and no opening some links I couldn't make you much sense of it. So I googled zabbix post authentication exploit, and first search result gave me reverse-shell exploit for zabbix 5.0.17. I confirmed that I was using the same version. On running this exploit file, I got the shell.

Owning The User

This phase was quite frustrating for me as I ran linpeas as spent few hours looking for some vulnerability until I found that user ipmi-svc had the same password as zabbix.

1su ipmi-svc
2cd
3cat user.txt

And I got the user flag!
